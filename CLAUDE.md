## 1. Что строим и зачем

Автономный торговый агент для **Polymarket** (рынок предсказаний). Пользователь задаёт сумму капитала и срок работы — бот сам анализирует лоты, принимает решения, исполняет их и отчитывается: **с чем торгует, на сколько и почему считает лот рентабельным**.

**Главный принцип проекта:**
> Сначала доказать, что ИИ-анализ реально бьёт рынок, и только потом рисковать реальными деньгами.
> Всё начинается в режиме **paper trading** (виртуальные сделки). Переход на реальную торговлю = переключение одного флага в конфиге, не переписывание.

Метрика успеха MVP — не красивый UI, а **таблица точности**: калибровка прогнозов модели против реальных исходов (Brier score, hit rate). Если edge не подтверждается на бумаге — реальные деньги не подключаем.

## 2. Архитектура (3 слоя, строго разделены)

```
[ЯДРО: торговый движок]      ← чистая Python-логика, без фреймворков
   fetcher → analyzer → risk_manager → executor
        ↓
[FastAPI]                    ← REST + WebSocket поверх ядра
        ↓
[React UI]  +  [aiogram*]    ← интерфейсы, дёргают одно ядро через API
                               (*Telegram опционально, как канал уведомлений)
```

**Железное правило:** торговая логика НИКОГДА не живёт в обработчиках Telegram/HTTP. Движок — отдельный модуль, интерфейсы — тонкие обёртки. Это гарантия отсутствия тупиков при росте проекта.

## 4. Слой анализа (ключевая логика edge)

Двухуровневая схема (тиринг для экономии и качества):

| Слой | Модель | Назначение | Цена (вход/выход за 1M токенов) |
|---|---|---|---|
| **Фильтрация** | Groq `llama-3.1-8b-instant` или `llama-3.3-70b-versatile` | Быстро прогнать ВСЕ лоты, отсеять неинтересные | бесплатный tier / $0.05–$0.59 |
| **Решение** | xAI `grok-4.3` | Глубокий разбор 3–5 финальных кандидатов | $1.25 / $2.50 |
| **A/B (опц.)** | Claude (Sonnet 4.6 / Opus 4.8) | Сравнение калибровки на своей статистике | $3/$15 — $5/$25 |

**Почему так:**
- Groq — дёшево/бесплатно и очень быстро, но только открытые модели (Llama и пр.).
- Grok 4.3 — дёшево, OpenAI-совместим (одна библиотека с Groq), есть нативный X/web-поиск для событийных лотов. **Важно:** у Grok knowledge cutoff ~ноябрь 2024 → для актуальных событий включать серверный Web/X Search (тарифицируется отдельно, ~$5 за 1000 вызовов).
- Claude — лучшая калибровка вероятностей, но отдельный SDK и дороже. Оставляем как кандидата, решаем по данным.

**Анализатор должен быть провайдеро-независимым:** интерфейс `analyze(market) -> {prob, verdict, reasoning, model}`, провайдер выбирается через конфиг. Это позволяет гонять несколько моделей на одних лотах и сравнивать их точность.

### Оптимизация стоимости (заложить с самого начала)
- **Промпт-кеширование** — системный промпт/методология одинаковы во всех вызовах → кеш экономит до ~90% на повторяющемся контексте (у Groq кешированные токены не считаются в рейт-лимиты).
- **Батчинг** — пакетная обработка дешевле на 50%.
- **Тиринг** — дорогую модель звать только для финалистов.
- Следить за **токен-капами** бесплатного Groq (на 70B ~100K токенов/день — легко упереться при объёме).

## 5. Polymarket API — что важно знать

Три раздельных API:
- **Gamma** (`gamma-api.polymarket.com`) — публичный, без авторизации. Поиск рынков, цены, метаданные. Отсюда берём кандидатов.
- **CLOB** (`clob.polymarket.com`) — торговля. Нужна авторизация: API-ключ выводится из кошелька + подпись ордеров по **EIP-712**.
- **Data API** — позиции и история пользователя.

**Критичные детали:**
- Сеть — **Polygon (chain id 137)**, расчёты в **USDC**.
- **НЕ путать `condition_id` (рынок) и `token_id` (конкретный исход Yes/No)** — торговля идёт по `token_id`.
- Есть **WebSocket** для real-time ордербука/цен.
- Движок исполнения Polymarket ориентирован на регион **eu-west-2** → см. деплой.
- **Геоблок:** Polymarket блокирует ряд юрисдикций. Торговый бэкенд разворачивать в ЕС.

## 6. Схема БД (PostgreSQL)

```sql
-- Рынки (лоты)
markets(
  id PK, condition_id UNIQUE, question, category,
  close_time, volume, status, url, fetched_at
)

-- Токены исходов (Yes/No и т.п.)
market_tokens(
  token_id PK, market_id FK, outcome, last_price
)

-- Решения анализатора (по каждому прогону)
analyses(
  id PK, market_id FK, token_id FK,
  model,              -- какая модель дала прогноз
  market_prob,        -- цена рынка на момент анализа
  my_prob,            -- оценка модели
  edge,               -- my_prob - market_prob
  verdict,            -- BUY_YES / BUY_NO / NEUTRAL
  reasoning,          -- текст обоснования (для отчёта пользователю)
  created_at
)

-- Позиции (виртуальные в paper, реальные в live)
positions(
  id PK, market_id FK, token_id FK,
  side, size_usdc, entry_price,
  mode,               -- 'paper' | 'live'
  status,             -- open | closed
  opened_at, closed_at, pnl
)

-- Исходы закрытых рынков
resolutions(
  id PK, market_id FK, winning_outcome, resolved_at
)

-- Точность считается из analyses + resolutions:
-- Brier score, hit rate, калибровка — по каждой модели отдельно
```

## 7. Риск-менеджмент (ХАРДКОД-предохранители)

Эти лимиты — не настройки «по желанию», а защита от слива банка. Зашить в `risk_manager`:

- **Макс. на одну позицию:** 3–5% от банка.
- **Макс. общая экспозиция:** напр. 50% банка одновременно в открытых позициях.
- **Стоп по просадке:** при −20% от стартового банка бот ОСТАНАВЛИВАЕТСЯ и зовёт человека (не продолжает торговать).
- **Минимальный edge для входа:** торговать только если |my_prob − market_prob| > порога (напр. 0.10). Нет edge — нет сделки.
- **Фильтр ликвидности:** игнорировать лоты с объёмом ниже порога.
- **Фильтр срока:** задаваемый диапазон времени до закрытия.
- **Spend-cap на LLM API:** жёсткий месячный лимит в консоли xAI + алерты (бот в цикле без лимита прожигает бюджет).

## 8. Paper vs Live (один флаг)

```python
TRADING_MODE = "paper"  # "paper" | "live"
```

- **paper:** `executor` пишет виртуальные сделки в БД, реальные ордера НЕ отправляются.
- **live:** `executor` вызывает `py-clob-client` с приватным ключом кошелька.

Логика анализа, риск-менеджмента и логирования идентична в обоих режимах. Переход на live — только после подтверждённой точности на бумаге (цель: ≥50–100 закрытых прогнозов, положительный edge после комиссий).

## 9. Развёртывание

- **Торговый бэкенд** (FastAPI + движок + кошелёк) → **VPS в ЕС** (Hetzner, Германия/Финляндия). Только он чувствителен к геоблоку Polymarket.
- **React-фронтенд** → можно где угодно (Vercel/Netlify) — геолокация ему безразлична, он просто стучится в API.
- **БД** → Neon/Supabase или на том же VPS.
- **Запуск:** Docker + docker-compose, процессы под systemd/в контейнерах.
- **Бесплатные PaaS** (Railway/Render/Fly.io) годятся для MVP — проверить регион.

## 10. Безопасность

- **Приватный ключ кошелька** — только в переменных окружения, НИКОГДА в коде/репозитории. Отдельный кошелёк под бота с ограниченной суммой, не основной.
- **API-ключи** — env vars; отдельный ключ на dev/prod; ротация; 2FA на аккаунтах провайдеров.
- **Spend-caps** на всех LLM-провайдерах.
- `.env` в `.gitignore` с первого коммита.

## 13. Дорожная карта

- **Фаза 0 — Каркас:** структура проекта, БД, конфиг, заглушки модулей ядра.
- **Фаза 1 — Данные:** `fetcher` тянет реальные лоты с Gamma API → в БД.
- **Фаза 2 — Анализ:** `analyzer` + провайдеры (Groq фильтр → Grok решение), пишет в `analyses`.
- **Фаза 3 — Paper-торговля:** `risk_manager` + `executor (paper)` + `scheduler`. Бот крутит цикл, пишет виртуальные сделки.
- **Фаза 4 — Учёт точности:** трекинг исходов (`resolutions`), расчёт Brier/hit rate по моделям. FastAPI отдаёт статистику.
- **Фаза 5 — UI:** React-дашборд (позиции, обоснования, кривая точности, P&L).
- **Фаза 6 — Live (только если edge подтверждён):** реальный кошелёк, малая сумма, `TRADING_MODE=live`.
